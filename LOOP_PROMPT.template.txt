# RESUME CHECK  (Opus — first turn only, before PLAN)
- If this worktree already has work in it (a row mid-flight, uncommitted changes),
  establish state BEFORE doing anything new: read LOOP_LOG.md + the roadmap to see
  where it stopped, then run the FULL test suite + build/lint. Is the tree green?
  If a prior cycle left green-but-uncommitted work, COMMIT it first so it can't be
  lost. Only then continue. Never build on top of an unknown/red tree.

# PLAN  (Opus — you, the main session)
- Read the roadmap. Find the FIRST row whose Status is not done/blocked/failed/
  in_progress. Set it to "in_progress" immediately (claim it) — prevents a
  parallel loop grabbing the same row.
- Re-read the goal/rule. Write a TIGHT, self-contained spec for this ONE row:
  what changes, which files, the EXACT tests that prove it (TDD), AND — if the
  row needs them — which tools to call (web, MCP, media, publish) and the
  REAL artifact it must produce (file written + non-empty, URL live, asset
  generated). One row, one PR, no scope creep. The spec is the entire handoff —
  the build subagent sees none of your context, only what you write.

# DO  (delegate to Sonnet — do NOT build inline yourself)
- Spawn a subagent (Agent tool, model "sonnet") with the spec verbatim.
  Test-first: write the failing tests, then the code to pass them, nothing
  outside the spec. If the row needs a tool (data fetch, asset gen, API call),
  the builder uses it (load MCP/web schemas via ToolSearch). It returns the
  diff + test output + the paths/URLs of any artifacts produced.
- You (Opus) stay out of implementation; your next turn is CHECK.

# CHECK  (Opus — you)
- FIRST verify the subagent actually did the work — do NOT trust its self-report.
  A subagent can return a confident "done — files created, tests pass" while
  having written NOTHING to disk (seen in the wild: tool_uses:1, clean git status,
  test count unchanged). Confirm against the filesystem yourself: `git status`,
  the new files exist, the test COUNT went up. If reality ≠ its report, redo the
  work directly or re-dispatch — never accept the claim.
- Then review the diff against the spec. Run the FULL test suite + the new tests.
  Run the linter/build YOURSELF (not the subagent's reported output). If a prod
  build fails on something that looks impossible (a route "not found" that exists),
  suspect a stale build cache — clear it (e.g. rm -rf .next) and re-run before
  believing the failure.
- Anything fails → hand the failure output back to Sonnet to fix, re-run. At
  most TWICE. Still failing → row "failed", reason in Notes, park the WIP, move
  on to the next row.

# VERIFY THE ARTIFACT  (Opus — you)  ← never skip. green tests ≠ working thing.
- Actually RUN the thing the row produces and confirm the real output exists.
  Run the build/generate/publish script and ASSERT the artifact is real:
  file present AND non-empty (row count > 0, HTML contains the expected
  content, asset file exists, API call returns 200). If the row only added a
  pure function, run the caller that uses it.
- A serializer with a passing unit test but NO emitted file is NOT done —
  generate the file and check it. (This is the exact gap that bit the first
  owenator run: 64/64 tests green, zero artifacts on disk.)
- If the artifact can't be produced (missing data/cred) → row "blocked", note
  why, move on.
- SPEC-DRIVEN rows (a row mapped to numbered acceptance criteria / FRs from a
  spec): also run an ADVERSARIAL spec-verifier — spawn a subagent (sonnet; opus
  for security/compliance criteria) with ~/playbooks/VERIFIER_PROMPT.template.txt:
  paste the criteria + how to exercise the build; it returns PASS/FAIL+evidence
  per criterion, defaulting to FAIL. Any FAIL → fix the gap, re-verify, until the
  whole list is PASS. This is the gate for anything with a real spec behind it.
- Runtime-verify gotchas: a dev server spawned inside a tool call can get reaped —
  run it as a tracked background task and poll for ready, or verify against a
  clean production start (npm build && start), not a stale dev server.

- BACK UP OFF-MACHINE (every shipped row, for redundancy): push to GitHub —

- SHIP per {{DEPLOY}}:
    git                → run the project's build/publish script so the live

    vercel-prod        → `vercel --prod`        (NOT a git-push; git deploys wedge)

- Set the row "done" with the sha + "shipped+verified" in Notes. Append one
  line to LOOP_LOG.md: timestamp, row #, outcome, sha, deploy result, approx
  tokens. Start the next cycle from PLAN.

# ROADMAP COMPLETE → 360 CAPSTONE  (when every row is done/blocked/failed)

# HANDOFF  (write HANDOFF.md whenever the loop stops — completion, hard stop, or
#           a pile-up of blocked/failed rows a human must clear)
- A dev who clones this branch gets the CODE from git. HANDOFF.md must carry the
  things git does NOT: the why, the dead-ends, the run/deploy gotchas, and what to
  do next. Do NOT restate the diff or the commit log — point at them.
- Write `HANDOFF.md` in the repo root, then commit + push it with the branch
  (loop-backup.sh / git push) so it travels off-machine. Sections:
  * STATE — counts (done/in_progress/blocked/failed) + every blocked/failed row
    listed with its one-line WHY (the reason from Notes). What works, what's stubbed.
  * LAND HERE — branch (loop/{{APP}}), worktree path, last sha, and the exact way
    to get this tree (git fetch + checkout the branch / worktree add).
  * RUN IT — env setup (which .env, the dev-DB host — NOT a prod URL), install, the
    test command, dev/build command, and the deploy command for {{DEPLOY}} incl. its
    trap (e.g. vercel-prod = `vercel --prod`, NOT a git push).
  * WHY / DECISIONS — the non-obvious calls made this session and any approaches
    TRIED AND ABANDONED (so the dev doesn't re-walk them). This is the part that
    never makes it into commits.
  * GOTCHAS — the project rule ({{RULE}}), plus traps hit this session (stale .next
    cache, sslmode=no-verify for the local DB, a flaky step, etc.).
  * NEXT ACTIONS — ordered punch-list a human picks up: unblock steps for each
    blocked row first, then the remaining todo rows. Say what to do FIRST.
- If a 360 capstone ran, link its SPEC_COVERAGE_360.md from NEXT ACTIONS rather than
  duplicating it. Keep HANDOFF.md current — overwrite it on each stop, don't append.

# HARD STOPS  (halt the loop, write HANDOFF.md, summarize state, do NOT continue)
- Cumulative spend ~1M tokens this session, or wall-clock 3 hours.
- DATABASE WRITES — protect PRODUCTION above all (the safety is about the TARGET,
  not the action):
  * A migration/backfill against a confirmed NON-prod DB (a local/docker dev DB or
    a branch) is FINE when the project rule authorizes it. Otherwise the default
    holds: row "blocked", STOP for human sign-off — never migrate autonomously.
  * PREFLIGHT — once per session, BEFORE the first migration or any write-heavy
    test: echo the DATABASE_URL host and CONFIRM it is the project's dev DB. A
    managed production host (neon.tech / rds / cloudsql / any prod hostname) is an
    automatic STOP — never run a migration or write test fixtures against it —
    UNLESS the rule names that exact prod step as deliberate + human-gated. A loop's
    .env.local must point at its dev DB BEFORE row 1 runs; never inherit a repo's
    prod DATABASE_URL. (This bit defacto-x: the loop migrated prod because its env
    still held the prod URL at launch.)
- A row needs a missing credential / external decision → row "blocked", SKIP to
  the next row (not a full stop).
- A row would WRITE to a read-only system (Finale, Metorik) → row "blocked".
- A row would send REAL outbound (email/SMS/ads/social posts) → row "blocked"
  for human review. Drafting into Gmail Drafts is fine; sending is not.
